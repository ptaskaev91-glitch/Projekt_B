# ADR 001 — Guardrails Against Premature Backend/Abstractions

## Context
- Фронт уже имеет Supabase/Horde интеграции; риск — добавлять новые backend/репозитории без чёткого доменного смысла.
- Прошлые проблемы: бизнес-логика в UI, нестабильные потоки действий.

## Decision
- Не создаём новый backend/сервис, пока нет сформулированной доменной сущности и потока использования (шаг 32).
- Любая новая персистенция начинается с описания доменной модели и намерений (шаг 33).
- Запрещены generic repository/factory абстракции без 3+ повторений конкретного паттерна (шаг 34–35).
- Каждое архитектурное ограничение фиксируется ADR; этот документ закрывает шаги 32–36.

## Consequences
- Быстрые фичи идут через существующие сервисы; новые хранилища — только после описания домена.
- Код остаётся конкретным; вынос в общие абстракции — только при подтверждённой повторяемости.
- Все участники опираются на единый набор правил, зафиксированных в ADR 001.
