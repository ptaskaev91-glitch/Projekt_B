# Правила работы

## Основные принципы
1. **Пишем коротко, делаем много**
2. **Лимит времени:** 5 минут на ответ, после 4 минут — логично завершаем и переносим на следующий этап
3. **Перед каждым этапом читаем:**
   - `rules.md`
   - `development.md`
   - `history.md`
   - `structure.md`
4. **После каждого этапа обновляем:**
   - `development.md`
   - `history.md`
   - `structure.md` (если менялась структура)
5. **Переписку чата не перечитывать** — только последние 6 сообщений при необходимости + `history.md`
6. **В начале работы фиксируем, что собираемся сделать, сверяемся со всеми 4 файлами .md**
7. **В конце каждого сообщения указываем, что сделано и что планируем дальше**
8. **UI не мутирует глобальное состояние напрямую — все новые действия через dispatch (Intent → Decision → State → Render)**
9. **Cross-component workflow запрещён** — компоненты не координируют друг друга напрямую, связь только через `dispatch` и общее состояние
10. **Новый backend/сервис только после описанного домена и потока намерений (ADR 001)**
11. **Любая новая персистенция начинается с доменной модели; без неё — нельзя**
12. **Repository/Factory абстракции — только при ≥3 повторениях конкретного паттерна**
13. **Каждое архитектурное решение фиксируется ADR (начиная с `adr/001-architecture-guardrails.md`)**

## Порядок работы
1. Читаю документацию (rules, development, history, structure)
2. Понимаю текущее состояние
3. Делаю максимум за 4 минуты
4. Проверяю сборку
5. Обновляю документацию
6. Кратко отчитываюсь

## Файлы документации
| Файл | Назначение |
|------|------------|
| `rules.md` | Правила работы |
| `development.md` | План, статус, что сделано |
| `history.md` | История переписки |
| `structure.md` | Структура проекта |
